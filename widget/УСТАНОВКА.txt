Виджет «ИНН → DaData» для карточки компании в amoCRM
==================================================

ГДЕ ОБНОВЛЯТЬ КОД (важно)
-------------------------
В **вашем аккаунте CRM** (Настройки → Интеграции) вы только **включаете** виджет и заполняете
**поля** (backend_url, x_api_key, field_*). Оттуда **нельзя** «подгрузить новый script.js» — так у
amo устроено; это не отсутствие функции в вашем интерфейсе, а разделение ролей.

Код виджета меняется только в **кабинете разработчика** (отдельный раздел для интеграций, не карточка
сделки): вход на сайте amo в раздел для разработчиков интеграций → **ваша интеграция** → редактор
файлов / проект виджета → вручную **вставить** новое содержимое `script.js`, `manifest.json` (скопировать
из папки widget/ в репозитории) → **сохранить и опубликовать** новую версию интеграции.

Если такого раздела у вас **нет** и интеграция только с **маркетплейса** — обновить чужой код нельзя;
нужен доступ владельца интеграции или **своя** новая приватная интеграция. Обойтись без обновления JS
можно: **робот + POST** на ваш `/integrations/amo/webhook` (см. ниже «Вебхук на Render»).

НЕТ КАБИНЕТА РАЗРАБОТЧИКА
-------------------------
Тогда **обновлять script.js в amo вы не можете** — ни с GitHub, ни с Render это не «подтянется» само.

Что реально сделать:
• **Основной сценарий:** робот в воронке (или другой триггер amo) → **вебхук** на ваш сервер
  `POST …/integrations/amo/webhook` с `lead_id` (как в шаблонах робота). Логика на **Render** в `main.py`
  — это и есть интеграция; виджет в карточке необязателен. Обновления — только **деплой** репозитория на Render.
• **Виджет в CRM:** останется той **версией**, что поставил владелец интеграции (маркетплейс / партнёр).
  Просить обновление у **автора интеграции** или оформить у amo **свою** интеграцию (если тариф/поддержка
  это позволяют — уточнять в поддержке amo / у менеджера).
• **Доступ к API amo** для сервера: long-lived токен / ключи в env на Render (как у вас уже настроено
  для webhook) — без кабинета разработчика клиентский код виджета всё равно не правится.

УЖЕ ЕСТЬ ИНТЕГРАЦИЯ (без загрузки ZIP)
--------------------------------------
ZIP не обязателен. В кабинете разработчика откройте свою интеграцию и:

• Подмените или смержите файлы из папки widget/ с теми, что в проекте интеграции:
  manifest.json, script.js, style.css, i18n/ru.json, images/*.
• В manifest в locations: lcard-1 (карточка СДЕЛКИ), comcard-1 (компания), settings.
  Настройки: backend_url, x_api_key, field_inn (ID поля ИНН на сделке, если есть),
  field_inn_company (рекомендуется — ID поля ИНН у компании; на карточке сделки ИНН
  чаще в блоке компании, без этого авто не сработает), остальные field_* — id полей.
• После сохранения опубликуйте/обновите версию интеграции в amo (как у вас принято).

Если в вашей интеграции один общий script.js — перенесите туда логику из нашего
script.js (fetch на …/company-by-inn, …/suggest-party и PATCH в amo) и пропишите
URL и X-API-KEY в ваших настройках / константах.

Подсказки при наборе: POST /suggest-party на бэкенде. На карточке СДЕЛКИ виджет при необходимости
сам вызывает GET /api/v4/companies/{id}, чтобы прочитать ИНН (во встроенной модели его часто нет).

Вкладка «Сеть»: по документации amo к внешнему API сначала идёт crm_post (прокси, в списке часто
только amocrm.ru); при ошибке — запасной fetch на backend_url (виден onrender). Раньше был обратный
порядок — из-за CSP fetch часто не доходил до Render. 1.2.6: на сделке при неудачном webhook или fields_updated=0
виджет сам запускает клиентский шаг (DaData + PATCH из браузера). developer_mode = 1: лог
«клиентский fallback после webhook». 1.2.7: при смене ИНН в карточке сделки сначала читается
модель в браузере (в т.ч. до кнопки «Сохранить»); только webhook видел бы старый ИНН в API amo.

Дальше — те же пункты про backend_url, x_api_key и ID полей (см. ниже).


ВАРИАНТ С ZIP (если в аккаунте есть загрузка архива)
---------------------------------------------------
1) Соберите ZIP из папки widget (в корне архива: manifest.json, script.js,
   style.css, папки images/ и i18n/).

2) Если amo даёт загрузить архив новой интеграции — используйте этот ZIP; если нет,
   ориентируйтесь на блок «УЖЕ ЕСТЬ ИНТЕГРАЦИЯ» (подмена файлов в проекте / у разработчика).


НАСТРОЙКИ
--------
3) В настройках виджета (или в коде) задайте:
   • backend_url — адрес вашего API БЕЗ слэша на конце, например:
     https://inn-efz1.onrender.com
   • x_api_key — тот же ключ, что и X-API-KEY на сервере (например test_key_1).

4) Поля field_inn, field_kpp, … — числовые ID дополнительных полей КОМПАНИИ в amo.
   Узнать ID: Настройки → Настройки компании → Поля → открыть поле,
   в адресе браузера будет …/detail/12345 — это field_id.
   Пустые строки — поле не заполняется.

5) Откройте карточку СОХРАНЁННОЙ компании (не новую пустую), введите ИНН,
   нажмите «Заполнить из DaData». Название компании подставится из DaData;
   остальное — в те поля, чьи ID вы указали.

6) На Render в Environment должен быть задан DADATA_API_KEY.

RENDER: Port scan timeout / no open ports
----------------------------------------
Uvicorn по умолчанию вешается на 127.0.0.1 — снаружи порт «не виден», деплой падает по таймауту.

• Start Command (обязательно 0.0.0.0 и переменная PORT):
    uvicorn main:app --host 0.0.0.0 --port $PORT
  Либо оставьте поле пустым — подхватится Procfile из репозитория (там то же самое).
• Не используйте: uvicorn main:app --port $PORT  (без --host 0.0.0.0).

RENDER: если в логах POST /suggest-party → 404
-------------------------------------------
На инстансе старый код или неверная команда старта.

• Settings → Build & Deploy: Root Directory пустой (корень репозитория).
• Start Command — см. блок выше (с --host 0.0.0.0).
• Manual Deploy → Clear build cache & deploy.
• Проверка: откройте в браузере https://ВАШ.onrender.com/
  в JSON должны быть "build":"suggest-party-v1" и путь "/suggest-party" в списке post.
  Если только {"status":"ok"} — деплой ещё не с актуального main.

Как проверить на ДРУГОЙ сделке
------------------------------
• Подсказки при вводе есть только в ПОЛЕ ВИДЖЕТА (блок справа), не в стандартном поле amo.
  Наберите 2+ символа (ИНН или слово из названия) — должен появиться список. Если под полем
  красное/серое сообщение об ошибке — читайте текст (часто 404 = старый виджет или старый деплой).
• Автозаполнение с карточки СДЕЛКИ срабатывает только если в настройках виджета field_inn =
  ID того же поля ИНН на СДЕЛКЕ, и в этом поле появились 10 или 12 цифр (amo должен сохранить
  изменение в модели). Если ИНН только у связанной КОМПАНИИ, а на сделке поля ИНН нет —
  откройте карточку КОМПАНИИ или введите ИНН в поле виджета и выберите из списка / «Заполнить».
• Быстрая проверка бэкенда: в браузере откройте https://ВАШ.onrender.com/ — в JSON должны быть
  "build":"suggest-party-v1" и "/suggest-party" в массиве post.

Вебхук на Render (без виджета) — не «одна сделка»
-----------------------------------------------
• В curl число 46873083 — просто пример. Подставьте ЛЮБОЙ id сделки: сервер не запоминает одну карточку.
• «Само» не обновится: Render не ходит в amo сам по расписанию. Нужен РОБОТ в amo, который при событии
  (создание сделки, смена этапа, сохранение полей — как настроите) делает POST на
  https://ВАШ.onrender.com/integrations/amo/webhook с телом вроде {"lead_id": "{{lead.id}}"}
  (макрос id сделки возьмите из подсказок шаблона вебхука в amo).
• Другая сделка «не заполнилась»: либо робот на неё не срабатывал, либо в момент вызова в ней ещё не было
  ИНН 10/12 цифр (ни на сделке, ни у связанной компании в API) — тогда ответ ok:false, reason BAD_INN.

Режим разработчика (виджет)
---------------------------
• Вызов webhook из виджета передаёт в JSON field_inn и field_inn_company из настроек — сервер читает ИНН
  по этим id (как в amo), а не только по AMO_FIELD_* на Render. Робот в воронке по-прежнему использует env.

• В настройках интеграции добавьте поле developer_mode = 1 (или true). В карточке появится блок лога и
  сообщения пойдут в консоль браузера (F12 → Console).
• После каждого POST /integrations/amo/webhook в логе видно: success, reason, fields_updated,
  updated_entity, lead_mirror_fields_updated — по ним проще понять, почему «авто» молчит.

Почему сделка не меняется по ИНН (чеклист)
-----------------------------------------
Уже исключено: публичный хостинг (Render) с интернета доступен; VPN на вашем ПК не мешает серверам amo
  достучаться до Render, если настроен робот с публичным URL.

1) Карточка СДЕЛКИ + заданы backend_url и x_api_key: авто идёт через webhook из браузера (не через робот).
   Не сработает, если не открыта сохранённая сделка, неверный ключ, или сервер вернул ok:false.

2) reason BAD_INN — в API сделки нет ИНН 10/12 цифр ни в поле сделки, ни у связанной компании
   (см. AMO_FIELD_INN / AMO_FIELD_INN_COMPANY на Render). Или ИНН ещё не успели сохранить в amo.

3) reason NO_MAPPED_FIELDS — на бэкенде не сопоставлены поля amo (переменные AMO_FIELD_* на Render).

4) ok:true, fields_updated > 0, но в интерфейсе старое — обновите страницу (F5); amo не всегда
   перерисовывает доп. поля без перезагрузки.

5) Тот же ИНН, что и в прошлый раз — виджет может не слать повтор (дедуп по последнему обработанному ИНН).

6) Робот в воронке шлёт webhook с серверов amo — без него «само по этапу» не запустится; curl с вашего
   ПК при этом всё равно может работать.

Подробнее про API бэкенда: см. main.py и /docs на вашем сервере.
